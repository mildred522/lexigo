# 主线收敛与基线整理设计

> 来源项目：`E:\aiproduct\安卓单词软件\content`
> 目标阶段：A（主线收敛）→ C（词库生产可交付化）→ B（Android 学习/复习功能）

## 1. 目标

本阶段的目标不是新增功能，而是把当前分散在 `master` 工作区与 `task-2-shared-models` 工作树中的有效成果，收敛成一个单一、可信、可继续开发的主线基线。

完成后，默认工作目录 `E:\aiproduct\安卓单词软件\content` 应当直接代表项目当前最新状态，后续 C、B 两个子项目都只在主线继续推进，不再依赖隐藏 worktree 才能理解或继续开发项目。

## 2. 当前问题

当前仓库存在明显的“默认入口落后、实际进展藏在 worktree”问题：

- `master` 工作区只保留了非常早期的 Python 项目骨架与最小测试。
- `task-2-shared-models` 工作树中已经存在：
  - 更完整的 Python 数据管线
  - 词库报告与验证脚本
  - 词库打包产物
  - Android 搜索与词库接入工程
- 这导致项目状态对外不可见，后续开发容易在错误基线上继续进行。

## 3. 范围

### 3.1 本阶段包含

- 盘点 `master` 与 `task-2-shared-models` 的有效差异
- 识别正式保留内容、噪音内容、待判断内容
- 将有效成果回填到主工作区
- 修正 README、脚本入口、文档路径与目录一致性
- 验证主工作区在 Python 侧的基线可用性
- 为下一阶段 C 建立稳定起点

### 3.2 本阶段不包含

- 新增 Android 学习模式功能
- 新增 Android 复习模式功能
- 重做词库数据架构
- 推进新的产品交互设计
- 大范围无关重构

## 4. 收敛原则

### 4.1 事实来源原则

以 `task-2-shared-models` 中“已通过测试、已形成产物、职责清晰”的内容为事实来源，而不是以旧 `master` 为准。

### 4.2 最小必要迁移原则

只迁移对主线继续开发有价值的内容，不把实验缓存、临时目录、构建中间物一并带回主线。

### 4.3 主线唯一可信原则

收敛完成后，`E:\aiproduct\安卓单词软件\content` 必须成为唯一默认入口。即使 worktree 仍然存在，也不再承担“真实最新状态”的职责。

### 4.4 为后续阶段服务原则

A 阶段的结果必须直接支撑：

- C：词库生产流程可交付化
- B：Android 学习 / 复习功能开发

如果某项内容不能帮助后续阶段推进，就不在本阶段扩展。

## 5. 保留与舍弃策略

### 5.1 应保留到主线的内容

- `src/`
- `tests/`
- `scripts/`
- `docs/`
- `android-app/`
- 必要的 `artifacts/` 中可复现、可解释、能支撑开发的产物

### 5.2 不应保留到主线的内容

- `.pytest_cache`
- 各类 `pytest-cache-files-*`
- `.gradle-local`
- `.android-local`
- 临时目录、人工实验目录、手工测试残留
- 构建中间产物
- 仅对某次本地实验有意义但不属于正式项目结构的文件

### 5.3 需谨慎处理的内容

- `artifacts/` 中的大文件
- 仅用于样本验证、但未来可能需要复现的中间结果
- `build/`、egg-info、打包残留

处理方式是优先保留“可复现逻辑”，而不是盲目保留“当前文件产物”。

## 6. 执行方案

采用以下方案：

### 方案 1（选定）

以 `task-2-shared-models` 为准，将有效成果回填到主线。

原因：

- 真实进展主要位于该分支 / 工作树
- 该路径已经具备测试、文档、脚本与 Android 工程的连续性
- 相比从旧 `master` 反向补齐，工作量更小，遗漏更少

### 方案 2（未选）

以 `master` 为准，手工挑拣 worktree 内容回填。

不选原因：

- 过慢
- 容易遗漏成果
- 容易重复劳动

### 方案 3（未选）

额外新建 integration 分支后再统一回主线。

不选原因：

- 对当前单人推进成本过高
- 增加流程重量，不解决核心问题

## 7. 预期目录状态

收敛完成后，主工作区应当呈现以下状态：

- 主目录中直接存在完整的 Python 管线、脚本、测试与文档
- 主目录中直接存在 `android-app/`
- README 与 docs 描述当前真实能力，而不是早期骨架
- worktree 中的关键能力已在主线可见
- 后续开发默认只进入主工作区

## 8. 验证标准

本阶段完成需满足以下条件：

1. 主工作区包含 `task-2-shared-models` 的关键有效成果
2. 主工作区不再依赖隐藏 worktree 才能理解当前项目状态
3. Python 测试在主工作区可运行通过
4. 主工作区存在与当前能力一致的 README / docs / scripts
5. Android 工程在主工作区具备连续结构，可继续开展 C、B 阶段工作

## 9. 风险与控制

### 9.1 风险：把噪音文件一并带回主线

控制：

- 先分类，再迁移
- 优先迁移源码、脚本、文档、测试
- 对缓存和中间物做明确排除

### 9.2 风险：误删仍有价值的实验成果

控制：

- 先做差异盘点
- 对可疑文件按“待判断”分类
- 优先保留可复现脚本，而不是一次性产物

### 9.3 风险：主线可见性恢复，但运行链路被破坏

控制：

- 迁移后立即做测试验证
- 检查关键脚本入口、文档路径和目录约定

## 10. 阶段结束定义

当主工作区已经成为唯一可信入口，并且 C 阶段可以直接在主工作区继续推进时，A 阶段结束。

下一阶段进入：

- C：词库生产流程可交付化
- 然后进入 B：Android 学习 / 复习功能
