# Android App Design

## Goal

Add an Android client inside the current repository that can consume the packaged local dictionary and provide a usable first-pass word lookup experience.

## Scope

This design covers only the first Android deliverable:

- create an Android app module in the current repository
- use Kotlin, Jetpack Compose, and a simple layered architecture
- install the packaged dictionary assets on first launch
- support local word lookup against the packaged SQLite database
- show a search page and a word detail page

This design does not include:

- account systems
- sync
- study plans
- favorites
- online translation requests from the app
- remote dictionary updates
- audio playback
- complex filtering

## Recommended Approach

Use a single Android app module with internal `data`, `domain`, and `ui` boundaries.

This is the fastest path to a working app while still preserving a structure that can be split into multiple modules later if the client grows. The app will not build dictionary data itself. It will consume the exported package generated by the existing data pipeline.

## Project Structure

Add a new Android project root under the current repository:

- `android-app/`
  - Gradle wrapper and Android build files
  - `app/`
    - Android application module
    - `src/main/assets/dictionary/`
      - `dictionary.db`
      - `manifest.json`
    - `src/main/java/.../data/`
    - `src/main/java/.../domain/`
    - `src/main/java/.../ui/search/`
    - `src/main/java/.../ui/detail/`

The existing Python feasibility and packaging pipeline remains unchanged in responsibility. Android only reads exported assets.

## Architecture

### UI Layer

The UI layer uses Jetpack Compose.

Initial screens:

- search screen
- word detail screen

Responsibilities:

- render current search state
- accept query input
- show loading, empty, and no-result states
- navigate from search results to detail view

### Domain Layer

The domain layer exposes app-facing use cases:

- `SearchWordsUseCase`
- `GetWordDetailUseCase`

Responsibilities:

- convert UI intent into repository calls
- keep search rules centralized
- isolate UI from raw SQL and schema details

### Data Layer

The data layer owns packaged dictionary access.

Core pieces:

- `DictionaryAssetInstaller`
- `DictionaryDatabaseProvider`
- `WordSearchDao`
- `DictionaryRepository`

Responsibilities:

- copy packaged dictionary assets into app-private storage on first launch
- read `manifest.json`
- open SQLite safely
- run exact and full-text queries
- map raw rows into app models

## Dictionary Consumption Strategy

The app consumes the exported package produced by the current pipeline:

- packaged database: `dictionary.db`
- packaged manifest: `manifest.json`

On first launch:

1. check whether the packaged database already exists in app-private storage
2. if not present, copy `dictionary.db` and `manifest.json` from assets
3. if present, compare install state against the packaged manifest and decide whether reinstall is needed

The app should treat the manifest as the stable contract, not inspect arbitrary pipeline files outside the package.

## Database Access Strategy

For the first version, prefer direct SQLite access over Room.

Reasoning:

- the existing package is already a concrete SQLite artifact
- the immediate goal is to ship the lookup loop, not redesign the schema
- direct access reduces adaptation risk for FTS-backed search

The code should still be shaped so a later Room migration remains possible:

- DAO-like query boundary
- repository abstraction
- app DTOs separated from raw cursor rows

## Search Behavior

The first release should support:

- exact lookup by lemma
- fuzzy lookup through FTS

Recommended runtime behavior:

- blank query: show idle state
- short or exact-looking query: run exact lookup first
- general query text: run FTS lookup
- merge and de-duplicate results if both strategies return records

Returned result fields should include:

- language
- lemma
- surface
- reading or IPA
- part of speech
- Chinese meaning
- source meaning text
- source name
- source entry id

## Screen Design

### Search Screen

The search screen is the first screen and the core MVP surface.

Layout:

- top app bar or simple title
- prominent search input
- content area below for stateful result rendering

States:

- idle state before search
- loading state while querying
- empty-result state
- populated result list

Each result item should show:

- lemma
- reading or IPA
- language
- Chinese meaning
- source meaning text as secondary support

### Detail Screen

The detail screen should show one selected word clearly.

Layout:

- large lemma title
- secondary reading or IPA
- language and POS chips or inline metadata
- Chinese meaning section
- source meaning section
- source metadata section

The detail screen is intentionally simple in the first version and should prioritize readability over feature density.

## UI Style Direction

Use a clean learning-tool style, not a developer-tool style.

Design constraints:

- light background
- restrained accent color
- strong text hierarchy
- word form visually dominant over metadata
- compact but readable list density

Avoid:

- dashboard-like cards everywhere
- noisy chrome
- filler-heavy layouts

## Data Flow

The first-pass runtime flow should be:

1. app starts
2. installer ensures packaged dictionary files exist in app-private storage
3. database provider opens the copied SQLite database
4. search UI sends query to the ViewModel
5. ViewModel triggers domain use case
6. use case calls repository
7. repository queries SQLite and maps results
8. UI renders result list
9. tapping a result opens detail view

## Error Handling

The Android app should explicitly handle these first-version failures:

- dictionary asset copy failure
- missing packaged manifest
- database open failure
- malformed or incompatible manifest
- empty query and no-result states

User-facing handling:

- show a readable initialization error if the dictionary cannot be installed
- do not crash on lookup failures
- show retry affordance where the failure is recoverable

Developer-facing handling:

- log installation and database-open errors
- keep database path and manifest parsing logic centralized

## Testing Strategy

### Unit Tests

- search use case behavior
- repository row-to-model mapping
- installer version/install decision logic

### Data Layer Tests

- prepackaged asset copy succeeds
- exact query returns expected fields
- FTS query returns expected fields

### UI Tests

- search input updates state
- results appear after lookup
- tapping a result opens detail screen

### Smoke Validation

- fresh install launches successfully
- packaged dictionary is copied successfully
- known term can be searched
- detail screen renders expected content

## Success Criteria

The first Android delivery is successful when:

- the repository contains a runnable Android app project
- the app installs and launches
- the app can query the packaged local dictionary
- the app shows a working search page
- the app shows a working detail page
- the app handles empty and no-result states without crashing

## Implementation Boundary

This Android phase should stop after the lookup loop is stable.

Only after that should later phases add:

- favorites
- study records
- examples
- remote package updates
- richer search controls
